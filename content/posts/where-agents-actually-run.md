Once you have written a few agents, the shape of an agent stops being interesting. It is always the same: a loop, some tools, some skills, some memory, an identity, one or more channels in, one or more hooks for events. What is interesting is where they run. The hosting story has not converged at all. There are at least eight distinct layers in production today with very different security postures, invocation models, and default state assumptions, and the gaps between them are more interesting than the layers themselves.

Let me walk through what's actually out there, with links so you can poke at any layer that catches your eye.

**System daemons and always-on services** are the thinnest layer in the taxonomy and the most architecturally interesting. [OpenClaw](https://github.com/openclaw/openclaw), [Hermes Agent](https://github.com/nousresearch/hermes-agent), [AIdaemon](https://davidloor.com/en/blog/how-openclaw-inspired-me-to-build-aidaemon), [CheetahClaws](https://github.com/SafeRL-Lab/cheetahclaws), and [Claude Code wrapped in `/loop`](https://www.contextstudios.ai/blog/claude-code-loop-autonomous-agent) install themselves as `systemd`/`launchd` user services with `loginctl enable-linger`. I am building [docker-agent-test](https://github.com/PaulKinlan/docker-agent-test) here too, with each agent as a Linux user in the `agents` group and inter-agent comms over local SMTP. The agent stays resident, listens to a chat surface, keeps memory across reboots. The reason the layer is thin is not a lack of demand. It is that running an LLM agent as your user means giving it your SSH keys and your filesystem, and that is a larger blast radius than most people are willing to accept yet.

**User-invoked CLIs** are the dominant power-user surface. [Claude Code](https://code.claude.com/docs/en/overview), [Codex CLI](https://openai.com/index/openai-on-aws/), [Gemini CLI](https://github.com/bradAGI/awesome-cli-coding-agents), [Aider](https://www.developersdigest.tech/blog/aider-vs-claude-code-2026-update), [Amp](https://thoughts.jock.pl/p/ai-coding-harness-agents-2026), [Pi](https://github.com/bradAGI/awesome-cli-coding-agents), Sketch.dev, Continue, and my own [agent-do](https://github.com/PaulKinlan/agent-do). The TTY is the channel, the working directory is the memory, identity is "whoever ran the shell". This is the layer most builders reach for first because it requires no ops. The limit is that the agent dies when the shell exits, which is exactly why people keep wrapping their CLI in `/loop` or `pm2` to lift it back into the daemon layer.

**Hosted HTTP services** are massively overserved in 2026. Every cloud has a managed-agents product. [Cloudflare Agents](https://developers.cloudflare.com/agents/) on Durable Objects with [Sandboxes GA](https://www.infoq.com/news/2026/04/cloudflare-sandboxes-ga/), [Anthropic Managed Agents](https://www.infoq.com/news/2026/04/anthropic-managed-agents/), [Bedrock Managed Agents powered by OpenAI](https://aws.amazon.com/bedrock/managed-agents-openai/), [Vercel Agents](https://northflank.com/blog/e2b-vs-vercel-sandbox), Modal, the [OpenAI Agents SDK](https://openai.github.io/openai-agents-python/mcp/). The differentiator stopped being "we host agents" months ago. It is now the [sandbox runner](https://northflank.com/blog/best-sandbox-runners) underneath, which is its own emerging market (E2B, Daytona, Modal, Vercel Sandbox, Northflank, Blaxel, Runloop), now mostly bring-your-own under Cloudflare's Containers.

**Email as a channel** is small but suddenly hot. [AgentMail](https://www.agentmail.to/) raised $6M in March 2026; [Lobstermail](https://lobstermail.ai/blog/the-real-complexity-of-giving-an-ai-agent-its-own-email-address-and-inbox) and AgentSend are adjacent; my own [emaila.gent](https://emaila.gent) and [mix.email](https://mix.email/) sit in this layer. The pattern is SMTP/IMAP as transport, threads as session memory, the email address as agent identity. The nice property is that every existing email client becomes an agent client for free. The under-served part is auth and forgery handling, because there is no convention yet for signed agent-to-agent email.

**The browser as a runtime** is the layer most relevant to anyone reading this who builds for the web. [WebMCP](https://thenewstack.io/webmcp-chrome-ai-agents/) shipped in [Chrome 146](https://developer.chrome.com/blog/webmcp-epp) and [Edge 147](https://patrickbrosset.com/articles/2026-02-23-webmcp-updates-clarifications-and-next-steps/). [Chrome Skills](https://blog.google/products-and-platforms/products/chrome/skills-in-chrome/) launched in April 2026. [WebLLM](https://webllm.mlc.ai/) and [WebextLLM](https://github.com/idosal/WebextLLM) push real LLM inference into the page via WebGPU. [Cloudflare Browser Run](https://blog.cloudflare.com/browser-run-for-ai-agents/) ships managed browsers for agents, with WebMCP support. The DOM is both UI and tool surface, the page is the channel, identity is the browser profile. The catch is persistence: background pages get killed under MV3, so the long-running loop story is poor.

**Editors and IDEs** are saturated. [Cursor](https://www.octavehq.com/post/windsurf-vs-cursor-vs-zed-which-ai-ide-in-2026), [Windsurf Cascade](https://petronellatech.com/blog/windsurf-codeium-ide-guide-2026), [Zed](https://medium.com/@kamilcollu/ai-code-editors-zed-cursor-and-windsurf-b8a068c9eea3), [GitHub Copilot agent mode and its cloud coding agent](https://docs.github.com/copilot/concepts/agents/coding-agent/about-coding-agent), Continue, Aider-in-editor. Copilot's cloud coding agent (issue → PR autonomously) is starting to blur the line between this layer and hosted HTTP, which is interesting and probably where editor-side agents end up.

**Chat-platform bots** are always-on agent loops, not rule bots. [Devin's Slack integration](https://www.cognition.ai/), [nanobot](https://github.com/HKUDS/nanobot), [Hermes' multi-channel gateway](https://github.com/nousresearch/hermes-agent), [LLM-slackbot-channels](https://github.com/Vokturz/LLM-slackbot-channels). Architecture is consistent: webhook or gateway WebSocket into a backend agent loop with tools and RAG.

**Multi-agent orchestrators** are where the patterns are still moving. [LangGraph](https://gurusup.com/blog/best-multi-agent-frameworks-2026) is now ahead of CrewAI in stars. [CrewAI Flows](https://dev.to/emperorakashi20/crewai-vs-langgraph-vs-autogen-which-multi-agent-framework-should-you-use-in-2026-5h2f) lean role-based and event-driven. [AutoGen / AG2](https://pratikpathak.com/langgraph-vs-crewai-vs-autogen-2026/) ships a GroupChat selector pattern. The [Anthropic Agent SDK orchestrator/subagent shape](https://www.anthropic.com/engineering/building-agents-with-the-claude-agent-sdk) is increasingly the reference.

Anything missing? Hit me up.

Three things jump out when I look at it.

The first is the daemon layer is genuinely thin and the bottleneck is security, not interest. People want a personal agent that lives on their box. Hermes, AIdaemon, OpenClaw, CheetahClaws, my own docker-agent-test are all proof of demand. Nobody has cracked the security story for "an LLM that runs as my user with my keys and my filesystem". That gap is the same gap Apple's `launchd` filled for system services in 2005, and nobody owns it yet. If you want to build something defensible in this field, that is where I would look.

The second is that hosted HTTP is so overserved that the differentiator has moved one layer down. Every cloud has a managed-agents product. The thing they are selling each other on is the [sandbox runner](https://northflank.com/blog/best-sandbox-runners). Whoever wins managed agents next year will win it because their sandbox is faster, cheaper, or more isolated, not because their agent abstraction is cleverer.

The third, and the one I keep coming back to, is that across all eight layers the convergent primitives are the same. Skills, tools, memory, identity, channels, hooks. You see the same shape implemented eight different ways. There is no manifest format that lets me take a Claude Code skill, run it as a Cloudflare Agent, and surface it inside a Chrome extension via WebMCP. That portable agent format is the missing piece across the entire taxonomy, and the thing I most want someone to design. I have a [related argument](/slice-for-agents) about what an agent at a URL should look like; the same shape applies one level up.

I have my own bet on which layer matters most over the next few years (it is the browser, for the boring reason that it is the runtime most people already have installed). But the more useful observation is that we already built the agent fabric we need. We just built it eight times and nobody named it.
