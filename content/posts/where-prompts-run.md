I got an email on Monday morning telling me what happened last week in the Chinese web development ecosystem. It wasn't a newsletter I subscribed to. It was my own agent reading the open web on my behalf, and emailing me a digest. The same inbox also has weekly notes on webauthn and passkey shipping, on OpenClaw, on a couple of other topics I'm tracking. I didn't write a scraper. I didn't wire up RSS. I just forwarded a handful of newsletters to an email alias I'd set up, told the alias "I'm interested in these topics, track them, tell me what's new each week," and that was it.

That alias is an app. Specifically, it's an app running on [mix.email](https://mix.email), a thing I built a month or so ago. Each alias is a program, the system prompt is the source code, and every sender is a user with their own persistent state. The digest I read on Monday was produced by running a prompt in a loop, with some tools, inside a sandbox, and then sending the output out as an email. That is the entire application. There is no front end. There is no user account. Email is the only interface.

I've been thinking a lot about [the prompt is the program](/prompt-is-the-program) since I wrote about it in February. That post was me figuring out the thesis. The projects I've shipped since are me figuring out the runtime. And the runtime question turns out to be the interesting one.

The shape of what I've been doing is pretty consistent. I start by writing down what I want the agent to do, in prose, in a markdown file. Then I build just enough plumbing around it to make the prompt runnable. Memory, tools, a loop, a way to pass in a task and get a result out. And then, almost always, I end up extracting that plumbing into something reusable. That extract is what I've come to think of as a harness.

[agent-do](https://github.com/PaulKinlan/agent-do) is the most literal version of that pattern. It started as the glue I was writing around every prompt-driven project and eventually hardened into a library: a provider-agnostic agent loop with first-class tools, skills, lifecycle hooks, memory stores, and a master/worker orchestrator. You pass in a model, a set of tools, a system prompt, and a task. The loop does the rest. It's small enough to read in an afternoon. I wrote it because I was copying the same fifty lines of setup into every new project and it felt dumb.

[emaila.gent](https://emaila.gent) is a different shape entirely. It's the observation that if you trust the prompt to be the program, the shortest possible distribution is an email address. You send mail to the service, it spins up a sandboxed agent, the agent reads, thinks, replies. No install. No account. No app store. The email is both the invocation and the protocol. I built it because I wanted to know whether the interface could be that minimal and still feel like using an app.

And [mix.email](https://mix.email) is what happened when I realised that the same shape generalises. If one email address can host one agent, a lot of email addresses can host a lot of agents, and each sender to each alias can have their own state. Each alias is a standalone app with an owner who defines what it does in natural language. Each person who writes to that alias gets a persistent sandbox. It's a platform where the programs are prose and the distribution is sending email. I got it live in a day because most of what makes it work was already in the harness.

Three projects, three different runtime decisions, one shared realisation. The prompt decides what the system does. The harness decides where it runs, and who can reach it, and what it can touch.

I thought for a while that I was early to this. I wasn't. The piece that broke that illusion for me was reading about [Pi](https://lucumr.pocoo.org/2026/1/31/pi/), Mario Zechner's minimalist coding agent. Pi's bet is that "LLMs are good at writing and running code, so embrace this" — it gives the model almost nothing. Four tools, the shortest system prompt of any agent I've seen. Then, once you have Pi, you build on Pi. OpenClaw is one of those things. OpenClaw is Peter's take on the same idea, using Pi components. snarktank's [clawchief](https://github.com/snarktank/clawchief) is a starter kit on top of OpenClaw that turns the whole thing into a founder's chief of staff by dropping in a pile of markdown policy files and some cron jobs. No code, just prompts and config.

There's a lineage there that I didn't know about when I started. Pi → OpenClaw → clawchief is one path. Mine (whatever the right name is for the agent-do to mix.email evolution) is a different one. [Karpathy's various small agents](https://github.com/karpathy) are another. [Steve Yegge's vibe coding](https://sourcegraph.com/blog/meme-driven-engineering), Rob Dodson's [second brain gardener](https://robdodson.me/posts/i-gave-my-second-brain-a-gardener/), Armin's [sketch](https://sketch.dev/) exploration of agent loops. Garry Tan's [gstack](https://github.com/garrytan/gstack) is 23 specialist skills stapled onto Claude Code. Each of these projects is a slightly different take on the harness shape, and the only thing they really share is the belief that prose is now a reasonable source format.

What the harness actually is, when you strip it down, turns out to be pretty simple. A loop. A way for the model to call tools. A place to keep state. A boundary around what the agent can see and do. Everything above that is opinion. The opinions are where the projects diverge. Pi believes in minimalism, so it refuses to ship MCP and expects the agent to write its own capabilities. agent-do believes in provider neutrality and ships with MCP mounting so the agent can borrow other people's tools. clawchief believes the policy documents should be the center of gravity, so the agent is thin and the priority-map / auto-resolver markdown files are thick. These are all reasonable positions. Nobody has the answer yet.

The part I'm most interested in right now is where the harness gets hosted, because I think that's where most of the change is about to happen. The obvious places are the ones I've already been using: a local shell, an email server, a sandboxed Deno worker. The less obvious ones are where things get fun.

I want `npx some-agent-thing-here` to be a real command. I want to type it and get a prompt-driven agent sitting in my terminal, with access to the files in my current directory, ready to run a task. I'm not special in wanting this. Claude Code already does most of it. Pi does a version of it. Codex, Gemini CLI, Amp, cursor, all of them live in the shell now. What I want is the ability to trivially publish new agents as npm packages. One command, one agent, one harness. The same way I publish a library. Because the thing I keep noticing is that I write a small agent for a specific job (review these PRs, summarise this inbox, triage these issues), and then I want to share it, and the distribution story is clumsy. If the prompt is the program and the harness is the runtime, the package manager should just know how to handle that.

The other host I think about a lot is the browser. I can't go into detail on what I'm personally building here, but the direction is not a secret. Chrome Skills shipped recently: a way to define repeatable flows in natural language that the browser can execute against web pages. Extensions already have hooks into every navigation, every DOM mutation, every user interaction. There's a latent orchestration surface sitting inside the browser that nobody has really built on top of yet. Imagine not one chat sidebar but a hundred small agents, each one listening for a specific kind of behaviour, each one able to nudge, suggest, or act.

Look at your own browser history for the last month. The patterns are obvious. You open the same tab at the same time every day. You search for the same kind of thing on Fridays. You follow the same click path through the same site most weeks. A proactive runtime could offer to do any of that for you, at the right moment, with one accept-or-dismiss prompt. That doesn't need a chat window. The chat window is a transitional UI. It's what you use when the system doesn't know what you want yet. The moment the system has enough context to propose next steps, the sidebar becomes the wrong shape. The right shape is the thing that already happens: a notification, a diff, an inline suggestion, a pre-filled form. Honestly, it's not even a new pattern. It's what the browser's address bar has been quietly doing for a decade.

I don't know if people want that. A lot of people hate anything that feels like the browser acting on its own. But I think the argument that "nobody wants a proactive browser" is the same category of argument as "nobody wants their phone to interrupt them" was ten years ago. We always say we don't want it, and then we spend five hours a day inside the thing that does it.

The other thing worth noticing about the browser is that it already has a pretty good packaging format for this kind of program. A Chrome extension is a `.crx` file, which is a zip containing a manifest, some HTML, some JavaScript, a few images. The manifest declares entry points and permissions. The service worker (in MV3) is event-driven, sandboxed, can't touch the DOM, can't call the network beyond the hosts listed in the manifest, can't evaluate inline scripts under its CSP, and wakes up on schedule when a `chrome.alarms` event fires. If you squint, that is an agent runtime someone already built. All you need to do is drop a prompt into the zip, point a generic loop at it, and let the browser enforce the boundary.

I put together a small demo to check whether this feels as clean as it sounds. It does, actually. The extension is called [prompt-in-a-box](https://github.com/PaulKinlan/prompt-in-a-box). Its whole implementation is about 500 lines of JavaScript. The behaviour is in `prompt.md`, which for the shipped example is a tab-hygiene agent: every 30 minutes, look at the user's open tabs, and if there are more than 20, send a desktop notification suggesting the oldest three to close. Swap `prompt.md` for a different set of instructions and the same extension becomes a different tool. The manifest decides what the prompt is allowed to reach (tabs, notifications, storage). The CRX format provides the sandbox. The alarm provides the schedule. The `chrome.storage` API provides the memory. I wrote almost no new infrastructure. Most of what makes it work is stuff the browser has been shipping for years.

That packaging question is the one I find most interesting right now. A CRX is clearly a sensible container for a single-user prompt-program. A mix.email alias is a sensible container for a shared-per-user one. A command-line npm package is a sensible container for a stateless one. But none of them are quite the same as each other. What we don't have yet is an agreed portable format: a zip, a manifest, a prompt, a capability declaration, that any of these hosts could accept. Extensions come closest. You could almost imagine a mix.email alias, a CLI agent, and a Chrome extension all loading from the same bundle and running the same prompt against whichever tools their host can safely provide. That is the format I most want someone to design.

Coming back to email for a second, because it's the host I underestimated the most. I wasn't trying to be clever with mix.email. I was trying to avoid building UI. Email turned out to be startlingly good as a runtime boundary. Everyone has it. It's already async. It survives prompt injection a lot better than a chat interface, because you can filter, sandbox, and self-validate a message before the agent ever trusts it. And it does something useful to my life right now, which the browser version still hasn't quite managed, which is: it shows up when I'm ready for it, not when it's ready for me. I read the Monday digest when I sit down with my coffee. I don't read it because a notification told me to. The agent does the work, the email waits.

The part I didn't expect, and still find a bit unsettling, is that these programs co-author themselves with the people using them. Every mix.email app has two layers: the master prompt the owner wrote, and the per-user state that accumulates as people interact with it. The master prompt says "this is a book club, track what each person has read, suggest their next book." The per-user state is the actual list of books Alice has read, the fact that Bob prefers short non-fiction, the two emails Carol sent last month explaining she only reads in French now. That state threads back into every subsequent invocation. It's not data the agent looks up on the side. It's part of the program. Run the same master prompt with Alice's context and it behaves one way. Run it with Carol's context and it behaves differently enough that you'd swear it was a different app.

That's a genuinely new shape. Traditional programs take inputs and produce outputs. Prompt-programs take inputs that rewrite the program before producing the output. The user's data isn't arguments, it's code. Which means every user's copy of the app is slightly different, and the owner's master prompt is less like a deployed binary and more like a seed that grows in soil the user provides. If the prompt is the program, the program is adaptive in a way that source code almost never is.

This is the angle I think gets underplayed in conversations about agents. People keep arguing about whether the chat interface is dead. The chat interface is fine. It just happens to be one of many. Email is a chat interface with inbound filtering and persistent threading. A CLI is a chat interface with line editing and return codes. Chrome Skills is a chat interface with DOM access. Each host shapes what the agent is good at, and different hosts are good at different things. Stop trying to pick the winner.

The wider question behind all of this, and the one I'm going to be talking about at Google I/O this year, is what the web looks like for people who mostly interact with it through agents. The short answer is: it will change in ways we're only just starting to feel. The parts of the web that are pure information, the stuff we scrape and re-present and summarise and digest, will increasingly be consumed by software long before a human sees it. That's already true for anyone using a research agent, or a price-compare bot, or the email digest I described at the top of this post. But the parts of the web that are tactile, that you feel in your fingers, that you want to pinch and zoom and rearrange, those aren't going anywhere. HTML Canvas, generative DOM, interactive components, rich browser primitives: all of these are getting better at the same time agents are getting better at reading static pages. Both directions get to win.

I keep meeting people who tell me that URLs are over, that the homepage is over, that the open web is over. I think they are going to be very conclusively wrong. The open web is what the agents are reading. The reason my mix.email agent can keep up with what's happening in the Chinese web dev scene, or with OpenClaw, or with the latest passkey implementations, is that people write this stuff down in public, on addressable URLs, in HTML I can trust. If you take URLs away you lose the training substrate for every agent in the world. I don't think that happens. I think the web gets weirder, not smaller.

What I'm watching for now, and what I think is worth other people watching too, is the convergence of three things: agent harnesses getting small enough and cheap enough to deploy anywhere, hosts (shell, browser, email, others I haven't thought of) each becoming a viable runtime for different classes of agent, and the web itself evolving a second layer for machine-consumption that doesn't erase the first layer for humans. None of those three are new ideas individually. Put them together and it's actually quite exciting.

I should go and reply to my email digest now. It asked me a follow-up question.
